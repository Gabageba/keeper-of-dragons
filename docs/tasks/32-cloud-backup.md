# 32 — Облачный бэкап сохранений (опционально)

**Фаза:** 8 — Мобилка и релиз · **Зависит от:** 04 · **Приоритет: низкий**

## Цель

Опциональный облачный бэкап одного JSON-снимка на игрока — чтобы не терять прогресс
при смене устройства/очистке кэша. Не мультиплеер, не лидерборды — просто резервная
копия. Можно отложить до пострелиза.

## Что делать

1. **Выбор бэкенда.** Firebase или Supabase (GDD). Минимально: анонимная авторизация
   + одна таблица/документ `saves/{uid}` с JSON-снимком и `updated_at`.
2. **Абстракция в SaveManager.** Добавить `SyncProvider` интерфейс:
   `pull(): SaveData | null`, `push(data)`. Локальное сохранение остаётся основным;
   облако — зеркало.
3. **Стратегия синхронизации.** «Last-write-wins» по `updated_at` (для соло-игры
   достаточно). При конфликте (облако новее локального) — спросить игрока «загрузить
   облачное?».
4. **Триггеры push.** При важных вехах и при pause/закрытии (дросселировать, чтобы
   не спамить запросы). Pull — при старте, если есть аккаунт.
5. **Приватность/безопасность.** Только свой снимок (правила доступа по uid). Никаких
   серверных проверок логики (читы не критичны — GDD).
6. **Оффлайн-устойчивость.** Нет сети — игра работает полностью локально; синк
   догоняет при появлении сети.

## Definition of Done

- [ ] Включение бэкапа создаёт анонимный аккаунт и заливает снимок.
- [ ] На новом устройстве вход подтягивает прогресс (last-write-wins).
- [ ] Конфликт версий разрешается с подтверждением игрока.
- [ ] Без сети игра работает; синк восстанавливается позже.
- [ ] Игра полностью играбельна и без облака (фича опциональна).
